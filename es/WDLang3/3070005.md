
## SocketChangeTransmissionMode( Función)



<a name="XUse"></a>
<a name="Use"></a>
<a name="description"></a>
Cambia el modo de transmisión utilizado en un enchufe. Esta función se utiliza para definir la codificación de los mensajes durante la transmisión.. [SocketRead](../WDLang3/3070014.md) y [SocketWrite](../WDLang3/3070002.md) están utilizando el modo de transmisión especificado.

**Atención**: El modo de transmisión debe ser idéntico en el servidor y en el ordenador cliente.

Observaciones:

- ![WINDEV](https://doc.pcsoft.fr/ext/images/us/WD.png) Si se utiliza una aplicación WINDEV en un servidor y en un ordenador cliente, el cambio de modo de transmisión debe realizarse en el ordenador cliente. **y** en el servidor.

- **UDP sockets**: Los UDP sockets son siempre "Socket without tag".. La función **SocketChangeTransmissionMode** no debe utilizarse con sockets UDP. El protocolo UDP es un protocolo poco fiable: la cadena devuelta por [SocketWrite](../WDLang3/3070002.md) puede no llegar o llegar varias veces. Las cadenas no siempre se envían en orden.


**Recordatorio**: Un socket es un recurso utilizado por las aplicaciones para comunicarse de un equipo a otro, independientemente del tipo de red.






<a name="Example1"></a>
<a name="sample_code"></a>

## Ejemplo


```wl
// Implement the transmission mode with EOF as end marker
IF SocketChangeTransmissionMode("Server", SocketEndTag) = True THEN
	Info("Transmission mode modified")
END
```


<a name="Example2"></a>



```wl
// Implement the transmission mode with CRLF as end marker
IF SocketChangeTransmissionMode(sSocketname, SocketEndTag, CRLF) = False THEN
	RESULT False
END
```

## Sintaxis

`<Result> = SocketChangeTransmissionMode(<Socket name> , <Type of transmission> [, <Option>])`
---

**`<Result>: (booleano)`**



- <u><u><u><u>True</u></u></u></u> si la función se ejecuta con éxito,

- <u><u><u><u>False</u></u></u></u> en caso contrario (nombre de socket no válido, por ejemplo). Si se produce un error, puede get más detalles sobre el error con [ErrorInfo](../WDLang1/3013008.md).




**`<Socket name>: (Cadena de caracteres)`**

Nombre del socket definido en el servidor. Este nombre fue definido:

- por [SocketCreate](../WDLang3/3070011.md) (para un servidor WINDEV simplificado).

- por [SocketCreateInfrarrojo](../WDLang3/3070012.md) (para un servidor WINDEV Mobile simplificado).

- por [SocketAccept](../WDLang3/3070004.md) (para un servidor estándar).

- por [SocketConnect](../WDLang3/3070016.md) (para un sitio WEBDEV).




![WINDEV](https://doc.pcsoft.fr/ext/images/us/WD.png) Atención: el nombre del socket distingue entre mayúsculas y minúsculas.

**`<Type of transmission>: (Constante)`**

Tipo de transmisión a utilizar:


|   |   |
| --- | --- |
| *SocketEndTag* | Se usa para especificar una etiqueta final en &lt;Opción&gt;.. <br>Se produce un error de WLanguage ("End Marker s not respected") si se reciben elementos después del end Marker. <br>Por Default, la etiqueta final es la cadena de caracteres "&lt;EOF&gt;".. <br>Mensaje en el siguiente formato: "Hola mundo&lt;EOF&gt;" |
| *SocketEndTagBuffer* | Se utiliza para especificar una etiqueta de extremo de búfer en &lt;Opción&gt;.. <br>Si se reciben elementos después del final Marker, la cadena recibida se lee hasta llegar al final Marker , la parte restante se almacena y se traslada a la siguiente operación de lectura. <br>Por Default, la etiqueta de fin de la memoria intermedia es la cadena "&lt;EOF&gt;". |
| *SocketNoEndTag* | No se agrega ninguna etiqueta a y/o se quita de la cadena transmitida. En este caso, el número máximo de bytes que se pueden transmitir se puede especificar con [SocketRead](../WDLang3/3070014.md). |
| *SocketSizeAtBeginning*<br>(Valor predeterminado) | Este mensaje es un búfer en el siguiente formato: <br><br><pre><code><Length>+CR+<String or buffer to send></code></pre><br><br>&lt;Largo&gt; es una cadena ANSI que corresponde a la longitud de la cadena o búfer en bytes.. <br>Ejemplo: "9"+CR+"HelloWord". |



**`<Option>: (Cadena de caracteres opcional o constante WLanguage (CR, CRLF, EOT))`**

Etiqueta final (por defecto, esta Marker es la cadena "&lt;EOF&gt;").

## Observaciones




### Tipo de transmisión de mensajes
<a name="tipo_transmision_mensajes_ELTPARAGRAPHE000168"></a>

El tipo de transmisión del mensaje especifica el modo utilizado para definir la longitud del mensaje.

**Por defecto** (*SocketSizeAtBeginning* constante), el número de caracteres del mensaje se especifica al principio del mensaje. Este modo de transmisión se recomienda cuando los sockets se utilizan para comunicarse entre dos aplicaciones WINDEV.

Las constantes *SocketEndTag* y *SocketEndTagBuffer* permiten utilizar el modo estándar para definir la longitud de los mensajes cuando se comunica por sockets. Este modo de transmisión se recomienda cuando los sockets se utilizan para comunicarse entre una aplicación WINDEV y otra aplicación.. En este caso, se debe incluir un carácter específico en el mensaje para indicar que el mensaje ha terminado.
<a name="NOTE0_2"></a>




### Diferencias de funcionamiento entre las constantes SocketEndTag y SocketEndTagBuffer
<a name="diferencias_funcionamiento_entre_las_constantes_socketendtag_socketendtagbuffer_ELTPARAGRAPHE000180"></a>

En ambos casos, la operación de lectura en el zócalo espera a recibir la etiqueta final.

- Con la constante ***SocketEndTag***: Se produce un error de WLanguage ("End Marker s not respected") si se reciben elementos después del end Marker.

- Con la constante ***SocketEndTagBuffer***: Si se reciben elementos después del final Marker, la cadena recibida se lee hasta llegar al final Marker , la parte restante se almacena y se traslada a la siguiente operación de lectura. Esto permite realizar operaciones de lectura de Partial. Este modo simplifica la Process de algunos protocolos estándar en INTERNET.
	Por ejemplo: al conectarse a un servidor de noticias (protocolo NNTP), el protocolo indica que la etiqueta final puede ser "&lt;CRLF&gt;" o "&lt;CRLF&gt;.&lt;CRLF&gt;". Con el modo "con buffer", puede leer la primera Line, y esperar la segunda etiqueta sólo si se espera una.



<a name="NOTE0_3"></a>




### Mensaje con etiqueta final
<a name="mensaje_con_etiqueta_final_ELTPARAGRAPHE000193"></a>

Si la codificación del mensaje tiene en cuenta la etiqueta final, no es necesario especificarla en el mensaje.
<a name="NOTE0_4"></a>




### Transmisión entre dos ordenadores que utilizan cadenas de caracteres en diferentes formatos (UNICODE y ANSI)
<a name="transmision_entre_dos_ordenadores_que_utilizan_cadenas_caracteres_diferentes_formatos_unicode_ansi_ELTPARAGRAPHE000201"></a>

Pueden ser necesarias algunas conversiones ([AnsiToUnicode](../WDLang1/3024002.md) o [UnicodeToAnsi](../WDLang1/3024025.md)) cuando los mensajes se transmiten entre dos ordenadores que utilizan cadenas de caracteres en formatos diferentes.
