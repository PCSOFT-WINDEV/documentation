## Presentación
<a name="presentacion_ELTTEXTE000707"></a>
WEBDEV ofrece varios métodos para probar sus sitios:

- prueba de todo el proyecto,

- prueba de una sola página,

- prueba de una sola consulta (véase la guía "Informes y Consultas" para más detalles),

- prueba de una sola reporte (ver la guía de "Informes y Consultas" para más detalles),

- la ejecución del proyecto paso a paso,

- prueba de rendimiento del sitio,

- test de regresión/test de automatic,

- prueba de estrés.




**La prueba de todo el proyecto** se utiliza para simular el inicio del sitio. Esto permite probar todo el sitio, incluso si su desarrollo aún no está completo. Tan pronto como se produce un problema, tienes la capacidad de iniciar el depurador para identificar y solucionar el problema.

**La prueba de una sola página** ejecuta sólo la página actual. Esto le permite ejecutar la prueba de su proyecto desde una página determinada o comprobar el modo de funcionamiento de una página tan pronto como su desarrollo haya terminado. Al igual que para la prueba del proyecto, el depurador se puede iniciar tan pronto como se produzca un problema.

**La prueba de una sola consulta** ejecuta sólo la consulta actual. Esto le permite comprobar el modo de funcionamiento de una consulta tan pronto como finaliza su desarrollo.

**La prueba de un solo reporte** ejecuta sólo el reporte actual. Esto le permite verificar la implementación de una reporte tan pronto como se desarrolla.. Al igual que para la prueba del proyecto, el depurador se puede iniciar tan pronto como se produzca un problema.

**Ejecutando el proyecto paso a paso** se utiliza para lanzar el depurador al iniciar el sitio. Esta solución permite controlar de cerca cómo funciona el sitio.

**El test de rendimiento** de su sitio le permite comprobar y optimizar su ejecución Time.

**El test de no regresión (o test automatic)** se basa en la ejecución de scripts. Le permite comprobar que, durante la ejecución de su sitio, ... las características existentes siguen siendo compatibles.

**La prueba de esfuerzo** se utiliza para iniciar varias conexiones simultáneas al mismo sitio dinámico de WEBDEV.

Además de estos métodos, WEBDEV también incluye la "cobertura de código" del sitio, es decir, la medición de la cantidad de código que se ejecuta cuando se prueba el sitio.

## Probar un proyecto WEBDEV
<a name="probar_proyecto_webdev_ELTTEXTE000731"></a>


### Probar el proyecto desde el editor
<a name="probar_proyecto_desde_editor_ELTPARAGRAPHE000057"></a>

Ejecutar la prueba desde el editor te permite probar:

- que el sitio presenta,

- el uso del sitio con diferentes navegadores.




La prueba de un proyecto se puede ejecutar independientemente del elemento actual en el editor.

Observación: Se puede elegir el navegador de prueba utilizado para ejecutar la prueba del proyecto:

- en las opciones de WEBDEV: en la pestaña "", en el grupo "", despliegue "" y seleccione "Opciones generales de WEBDEV". El navegador se puede seleccionar en la pestaña "Web".

- en las opciones del modo de prueba: en la pestaña "", en el grupo "", despliegue "" y seleccione "Navegador de prueba".




**Diferentes tipos de pruebas**

**Para ejecutar la prueba de un sitio estático desde el editor**:

1. En la pestaña "", en el grupo "", despliegue "" y seleccione "Depurar proyecto desde la página de inicio".

2. El editor se minimiza automáticamente.

3. Se abre el navegador especificado en las opciones de WEBDEV y se muestra la página de inicio del sitio.




**Para ejecutar la prueba de un sitio dinámico (Sesión o AWP) desde el editor**, hay varios métodos disponibles:

- En la pestaña "", en el grupo "", despliegue "" y seleccione "Depurar proyecto" (o presione Ctrl + F9).

- Haga clic en ![](https://doc.pcsoft.fr/en-US/images/image.awp?langid=7&name=ico_GO_Projet_WB_bl.gif) entre los botones de acceso rápido.




El editor se minimiza automáticamente, se abre el navegador especificado en las opciones de WEBDEV y se muestra la primera página del sitio.

**Para probar un sitio estático + dinámico (Sesión o AWP) desde el editor**:

- ***para probar la parte estática del sitio***: realizar las operaciones correspondientes a la prueba de un sitio estático.

- ***para probar la parte dinámica del sitio (Sesión o AWP)***: realizar las operaciones correspondientes a la prueba de un sitio dinámico.




Sitio dinámico de **(modo de sesión o AWP): Start**

Los siguientes módulos se inician automáticamente durante la prueba de un sitio WEBDEV dinámico (modo Sesión o AWP):

- El servidor Web instalado en el equipo y configurado para WEBDEV al instalar WEBDEV.
	La prueba no se puede ejecutar si no se inicia el servidor Web.

- El administrador de WEBDEV (WD28 0ADMIN.EXE).
	El administrador se utiliza para gestionar las conexiones con el servidor web y para configurar los sitios WEBDEV.
	**Observación**: se puede ejecutar una prueba del proyecto desde la página de pruebas del administrador (pestaña "Avanzado" de WD28 0ADMIN, botón "Página de pruebas").

- El motor WEBDEV (WD28 0AWP.EXE).
	El motor de WEBDEV se utiliza para gestionar las solicitudes de los usuarios de la web desde su navegador y para devolver la correspondiente página HTML dinámica. 
	**Observación**: El motor de WEBDEV se inicia sólo si el proyecto contiene páginas dinámicas.

- El navegador INTERNET.
	El navegador INTERNET muestra las páginas HTML del sitio WEBDEV.



<a name="NOTE2_2"></a>


### Probar el proyecto desde el administrador de WEBDEV
<a name="probar_proyecto_desde_administrador_webdev_ELTPARAGRAPHE000173"></a>

Ejecutar la prueba desde el administrador de WEBDEV (WD28 0Admin) le permite hacer la prueba:

- las características del sitio.

- las características propias de la Web (uso de cookies, etc.).




**Observación**: El administrador de WEBDEV sólo puede ser usado para probar sitios dinámicos (Sesión o AWP) o la parte dinámica de los sitios estáticos + dinámicos.

Ejecutar la prueba desde el administrador de WEBDEV equivale a iniciar el sitio dinámico desde un ordenador remoto.

Antes de desplegar un sitio de WEBDEV, recomendamos que ejecute la prueba de este sitio al menos una vez desde el administrador de WEBDEV.

**Para ejecutar la prueba desde el administrador de WEBDEV**:

1. Inicie el administrador de WEBDEV: en la pestaña "", en el grupo "", haga clic en "".

2. En la pestaña "Avanzado" del administrador de WEBDEV, haga clic en el botón "Página de prueba".


**Para detener el test**, vaya al administrador de WEBDEV (haga clic en 28 en la barra de tareas) y haga clic en "Desconectar" (pestaña "Conexiones").

**Observación**: El administrador de WEBDEV también le permite ejecutar una prueba de proyecto equivalente a la prueba de proyecto ejecutada desde el editor:

1. Inicie el administrador de WEBDEV: en la pestaña "", en el grupo "", haga clic en "".

2. En la pestaña "Connection", seleccione el sitio y haga clic en el botón "Test".

### Prueba de estrés/prueba de regresión
<a name="prueba_estresprueba_regresion_ELTPARAGRAPHE000231"></a>

WDTestSite se utiliza para realizar pruebas de estrés: WDTestSite se utiliza para iniciar varias conexiones simultáneas a un sitio WEBDEV dinámico (Session o AWP).

Cada Connection realiza una set de acciones en el sitio WEBDEV (escenario preset).

## Probando un sitio WEBDEV remoto (en modo AWP o Sesión)
<a name="probando_sitio_webdev_remoto_modo_awp_sesion_ELTTEXTE000767"></a>
Se pueden utilizar varios métodos para realizar una prueba y depurar un sitio en la computadora de desarrollo. Sin embargo, en algunos casos, es posible que tenga que depurar el sitio directamente en los equipos de los usuarios.

Desde su oficina en Londres, tiene la posibilidad de depurar un sitio que se ejecuta en un servidor Web en Taiwán.. La operación de depuración se realiza sin tener que ir a ninguna parte, en la configuración final directamente.

Hay dos características disponibles:

- Iniciando y depurando el sitio en un servidor de aplicaciones remoto.

- Depurando un sitio actualmente usado en un servidor de aplicaciones remoto.




Para estas dos características, se requiere una configuración específica para el ordenador remoto.

## Probar una página
<a name="probar_una_pagina_ELTTEXTE000791"></a>


### Probar la página desde el editor
<a name="probar_pagina_desde_editor_ELTPARAGRAPHE000272"></a>

**Para probar una página del editor**:

1. Abra la página para ser probada.

2. Haga clic en ![](https://doc.pcsoft.fr/en-US/images/image.awp?langid=7&name=ico_Go_Page_bl.gif) entre los botones de acceso rápido del menú WEBDEV. También puede pulsar F9.

3. El editor se minimiza automáticamente y se ejecuta la página.




Durante la prueba, se pueden ejecutar todas las funciones de la página. Tendrá la posibilidad de abrir otras páginas, por ejemplo.
<a name="NOTE4_2"></a>


### Detener la prueba de página
<a name="detener_prueba_pagina_ELTPARAGRAPHE000289"></a>

Se pueden utilizar varios métodos para detener la prueba:

- **método 1**:
	Cerrar la página que se está probando. WEBDEV muestra el editor utilizado al principio de la prueba.

- **método 2**:

	- Vuelve al editor con la barra de tareas o presiona Alt + Tab.

	- Confirmar que la prueba debe ser detenida. WEBDEV muestra el editor utilizado al principio de la prueba.

## Trazar un proyecto
<a name="trazar_proyecto_ELTTEXTE000821"></a>


### Principios de depuración
<a name="principios_depuracion_ELTPARAGRAPHE000308"></a>

La depuración de una aplicación consiste en:

- comprobando el modo de funcionamiento de una Process,

- entender el modo de funcionamiento de un programa existente,

- comprobar el valor de las variables,

- comprobar el modo de funcionamiento de los casos especiales en una aplicación o en un sitio.




El depurador se utiliza para realizar estas operaciones.

**Observación**: WEBDEV también incluye varias herramientas de rastreo (ventana de rastreo, cuadro de información, etc.). Para obtener más información, consulte ["Depuración sin depurador"](#NOTE5_4).
<a name="NOTE5_2"></a>


### Visión general del depurador
<a name="vision_general_del_depurador_ELTPARAGRAPHE000329"></a>

El depurador se utiliza para rastrear los programas de WLanguage con el fin de ayudar a mejorar estos programas.

La ejecución del código fuente se visualiza en la pantalla. Los procesos que se ejecutan se clasifican en orden jerárquico en el panel "Depurador".

Se puede ver el valor de las variables:

- individualmente en la punta de la herramienta de cada Variable.

- en el panel "Depurador".



<a name="NOTE5_3"></a>


### Características del depurador
<a name="caracteristicas_del_depurador_ELTPARAGRAPHE000345"></a>

El depurador está acostumbrado:

- averigua la llamada Stack,

- ver el contenido de las variables o expresiones,

- ejecutar el código paso a paso con la capacidad de saltar los bloques,

- usar puntos de ruptura condicionales,

- modificar el código mientras continúa la ejecución,

- ...



<a name="NOTE5_4"></a>


### Depuración sin depurador
<a name="depuracion_sin_depurador_ELTPARAGRAPHE000361"></a>

En algunos casos, la ejecución de un programa con o sin depurador puede ser diferente.

De hecho, el depurador introduce pausas en la ejecución del Process durante las cuales WINDEV realiza varias tareas.

Por lo tanto, el depurador no puede ser usado en una Procedure llamada por una timer o en el código de una barra de desplazamiento.

**Observación**: Consulte la ayuda en línea para obtener más detalles sobre los límites del depurador.

Para depurar estas aplicaciones, es posible que desee seguir los cambios de un valor, cómo se llaman los diferentes procedimientos, etc.

Esta información puede ser:

- que se muestra en la pantalla

- almacenados en un archivo de rastreo.




**Atención**: Si la información se muestra en la pantalla, sólo debe mostrarse durante las pruebas de aplicación.

**Mostrando información**

Se pueden utilizar dos herramientas para visualizar la información:

- ***cajas de información***: [Info](../WDLang1/3021011.md).
	Atención: Cuando se muestran, los cuadros de información bloquean la aplicación.

- ***la ventana de rastro***: [rastro](../WDLang1/3013050.md).
	La ventana de seguimiento aparece en el Corner superior izquierdo de la pantalla, sin interrumpir el programa.




**Gestionar la visualización de la información de depuración**

Mostrar la información de depuración en la pantalla es useful sólo en modo de prueba.

Cualquier pantalla inadecuada debe ser removida antes de distribuir una aplicación.

Para evitar cualquier descuido, se recomienda gestionar cómo se muestra la información de depuración a través de un Procedure global.

Por ejemplo:

```txt
PROCEDURE MyTrace(StringToTrace)
IF InTestMode() THEN
		Trace(StringToTrace)
END
```

En este código, dependiendo del resultado de [InTestMode](../WDLang1/3013011.md), la ventana de rastreo aparece sólo durante la prueba de aplicación.

Tal Procedure permite dejar la llamada a las ventanas de rastreo en el código de la aplicación sin ningún riesgo de mostrarla en las computadoras de los usuarios finales.

La llamada a este rastro Procedure es idéntica al uso de [rastro](../WDLang1/3013050.md):

```txt
MyTrace("Customer: "+...
  Customer.CustomerNum)
```


**Creando un archivo de rastreo**

Durante los procesos largos (procesos por lotes, etc.), para comprobar el modo de funcionamiento del programa, hay que guardar una traza física de los procesos ejecutados (un archivo de texto, por ejemplo).

La siguiente Procedure se utiliza para gestionar la visualización de la traza:

- en la pantalla (parámetro /DEBUG en el comando Line).

- en un archivo de texto (modo Default).
	
	```txt
	PROCEDURE MyTrace(StringToTrace)
	FilePath is int
	FilePath = fDataDirUser() + ProjectInfo(piProjectName)+ ".txt"
	FileNum is int
	DebugMode is boolean = False
	IF Position(CommandLine(), "/DEBUG") > 0 THEN
		DebugMode = True
	END
	IF DebugMode THEN
		Trace(StringToTrace)
	ELSE
		FileNum = fOpen(FilePath, foCreateIfNotExist + ...
	       foWrite + foAdd)
		If FileNum <> -1 THEN
			DateTimeTrace is DateTime
			DateTimeTrace = SysDateTime()
			DateTrace is Date
			DateTrace = MyDate..Date
			TimeTrace is Time 
			TimeTrace = MyDate..Time
			fWriteLine(FileNum, DateToString(DateTrace) + ...
				" - "+TimeToString(TimeTrace))
			fWriteLine(FileNum, StringToTrace)
			fWriteLine(FileNum, " ")
			fClose(FileNum)
		END
	END
	```



**Observaciones**:

- El archivo de seguimiento es creado por Default en el directorio de datos del usuario. Este archivo se llama como el proyecto. Este fichero contiene la información a rastrear durante la ejecución del programa.. La información se completa con la fecha y Time de cada "Rastreo". Esto le permite detectar un posible problema durante la Process.

- Ejemplo de contenido del archivo de rastreo:
	
	```txt
	01/12/2015 - 10:53:25:20
	Customer name: Montgomery
	```

## Prueba de rendimiento
<a name="prueba_rendimiento_ELTTEXTE000863"></a>


### Presentación
<a name="presentacion_ELTPARAGRAPHE000448"></a>

El perfil de rendimiento le permite comprobar y optimizar la ejecución de Time de su sitio.

Su principio es sencillo:

- Pruebe su sitio.

- Durante esta prueba, el perfilador de rendimiento hace un seguimiento de todas las acciones realizadas y los procesos correspondientes ejecutados.


Al final de la prueba, el perfil de rendimiento muestra:

- las 10 operaciones que consumen más Time .

- todas las acciones realizadas en el sitio probado, clasificadas por Duration (de la acción más larga a la más corta).




Puede seleccionar una Process para analizar las razones de su Process ing Time para optimizarla.
<a name="NOTE6_2"></a>


### Inicio del perfil de rendimiento
<a name="inicio_del_perfil_rendimiento_ELTPARAGRAPHE000471"></a>

Para iniciar el analizador de rendimiento, en la pestaña "", en el grupo "", despliegue "" y seleccione "Analizar rendimiento".

El proyecto se ejecuta automáticamente en modo de prueba. La Process para optimizar puede ser ejecutada en su sitio.

Para volver al editor, todo lo que tienes que hacer es cerrar tu aplicación o tu sitio.

Luego, el perfilador de rendimiento muestra el resultado del análisis.

**Observación**: El perfil de rendimiento debe utilizarse para optimizar su sitio (antes de que se distribuya, por ejemplo).
<a name="NOTE6_3"></a>


### Lectura de los resultados del perfilador de rendimiento
<a name="lectura_los_resultados_del_perfilador_rendimiento_ELTPARAGRAPHE000497"></a>

El perfil de rendimiento presenta el resultado del análisis en varias pestañas:

- la pestaña "Resumen" presenta los diez procesos más largos.

- la pestaña "mapping" presenta una vista gráfica de los principales procesos.

- la pestaña "Detalles" presenta todos los procesos que se ejecutan durante la prueba de la aplicación (desde el más lento hasta el más rápido).

- la pestaña "Llamadas" se utiliza para ver los detalles de las operaciones realizadas en un Process.


La siguiente información se muestra para cada Process:


|   |   |
| --- | --- |
| Función | Funcionamiento, Process o Procedure corren. |
| Tiempo Total | Ventana de ejecución de la función. |
| Núm. de llamadas | Número de llamadas realizadas a la función (Procedure o Process). |
| Tiempo llamada 1 | Tiempo de ejecución de una llamada a la función (Procedure o Process). |
| código % | Porcentaje de código ejecutado fuera de la llamada a una función de WLanguage o fuera de la llamada a una función personalizada o Procedure. |
| Padre | Process que llamó a la función. |


**Observación**:

- El pie de foto de "Ejecución completa" representa la cantidad total de Time para la ejecución de la prueba del sitio con el perfil de rendimiento.

- El epígrafe "Total de la página XXX" representa la cantidad total de Time para la ejecución de la página XXX (desde su apertura hasta su cierre).



<a name="NOTE6_4"></a>


### Elegir un Process para optimizar
<a name="elegir_process_para_optimizar_ELTPARAGRAPHE000544"></a>

El Process a optimizar se elige según varios criterios:

- ***la ejecución Process Time***. Los procesos más largos deben necesariamente ser optimizados.

- ***el porcentaje de tiempo dedicado a la Process de la función o Procedure***. Cuanto mayor sea este porcentaje, mayor será el número de procesos que se pueden optimizar en el código.


**Observación**: Si el Process corresponde a una función WLanguage, es bastante difícil de optimizar.

## Pruebas de regresión
<a name="pruebas_regresion_ELTTEXTE000905"></a>


### Presentación
<a name="presentacion_ELTPARAGRAPHE000560"></a>

Existen varias herramientas de prueba disponibles para garantizar la calidad de sus aplicaciones:

- El modo de prueba (proyecto Go o página Go) se utiliza para comprobar inmediatamente una modificación realizada en su sitio.

- WDTestSite que se utiliza para realizar diferentes pruebas en un sitio WEBDEV.


Para automatizar estas pruebas y aumentar la calidad de sus aplicaciones, tiene la capacidad de ejecutar pruebas de la unidad **automatic**. Estas pruebas se utilizan para comprobar fácilmente todas las características propuestas por sus aplicaciones.
<a name="NOTE7_2"></a>


### Pruebas automatizadas
<a name="pruebas_automatizadas_ELTPARAGRAPHE000574"></a>

Cada prueba contiene un escenario que se puede tratar directamente en la interfase del producto.. Este escenario está escrito en WLanguage y puede ser modificado en cualquier Time.

Estas pruebas pueden realizarse antes de cada implementación para comprobar el modo de funcionamiento de un sitio después de varias modificaciones.

Se pueden comprobar los siguientes elementos:

- los conjuntos de procedimientos

- las clases


Cada prueba es associated con un código de lenguaje WLanguage: el escenario de prueba. Este escenario se puede ver en el editor de código. El código de las pruebas puede ser modificado.

Las pruebas y el código de associated no se distribuyen a los usuarios finales. Por lo tanto, el número de pruebas no influye en el tamaño del sitio disponible para los usuarios finales.


<a name="NOTE7_3"></a>


### WDTestSite
<a name="wdtestsite_ELTPARAGRAPHE000604"></a>

WDTestSite se utiliza para ejecutar diferentes pruebas en un sitio WEBDEV.

Las siguientes pruebas pueden ser realizadas por WDTestSite:

- ***Prueba de esfuerzo***:
	La prueba de estrés consiste en simular la Connection de varios usuarios de la web a un sitio WEBDEV. Cada usuario de la Web ejecuta una set de operaciones (escenario) simultáneamente.

- ***Prueba de regresión***:
	El test de regresión consiste en comprobar el modo de funcionamiento de un sitio WEBDEV entre dos actualizaciones. La prueba de regresión consiste en comprobar si un escenario realizado con una versión anterior del sitio sigue funcionando correctamente una vez que se ha actualizado el sitio.

- ***Prueba de un sitio en modo multiusuario***:
	La prueba de un sitio en modo multiusuario se utiliza para comprobar si los accesos simultáneos a los archivos de datos se gestionan correctamente.. Esta prueba consiste en simular la Connection simultánea de varios usuarios de la Web a un sitio WEBDEV. Cada usuario de la Web ejecuta una set de operaciones (escenario) simultáneamente.

- ***comparison de diferentes servidores***:
	WDTestSite se utiliza para comparar la velocidad de diferentes servidores. Para ello, ejecute el mismo escenario en diferentes servidores y compare la ejecución Time de este escenario.

- ***Optimización de los procesos desarrollados en WLanguage***:
	WDTestSite se utiliza para comparar la ejecución Time de un escenario antes y después de que el código WLanguage fuera optimizado.
