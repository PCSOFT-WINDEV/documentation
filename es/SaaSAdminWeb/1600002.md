
## Administrador de SaaS - Añadir un sitio
			



## Presentación
<a name="presentacion_ELTTEXTE000126"></a>
La adición de un sitio SaaS se utiliza para declarar un sitio SaaS desplegado en un servidor de aplicaciones WEBDEV a un servicio Web SaaS con el fin de permitir su uso.

Esta operación debe realizarse después de la implementación del sitio.

**Observación**: Esta operación equivale a la llamada a [Sitio web de SaaSAdminAddSite](../WDLang3/1000019102.md).







## ¿Cómo proceder?
<a name="¿como_proceder_ELTTEXTE000156"></a>
Para añadir un sitio SaaS, en el menú principal del Administrador de SaaS, seleccione la opción "Sitios .... Añadir un sitio".

**Recordatorio**: El administrador de SaaS se encuentra en el servidor de aplicaciones que alberga el servicio web SaaS que controla el sitio.. 

Se solicitan los siguientes elementos:


|   |   |
| --- | --- |
| **Elemento** | **Rol** |
| *Nombre del sitio* | Este es el nombre implementación del sitio WEBDEV.<br>Este nombre fue dado durante la Procedure para el despliegue del sitio en la instalación asistente.<br>Por Default, es el nombre del proyecto.<br><br>**Observación**: Este parámetro distingue entre mayúsculas y minúsculas. |
| *Modo de gestión de los datos* | Hay varias opciones disponibles:<br><br>\ Modo de gestión de los datos \ Beneficios \ Inconvenientes \<br>\ --- \ --- \ --- \<br>\ Ninguna base de datos \   \   \<br>\ Una única base de datos compartida por todas las cuentas de clientes \ <br><br>- Capacidad para compartir elementos entre los diferentes clientes (si el sitio utiliza grandes memos, no es necesario copiarlos para cada cliente).<br><br>- Mantenimiento más fácil de la base de datos: una sola copia de seguridad para realizar, una sola modificación de automatic si el análisis del sitio evoluciona.<br><br><br> \ <br><br>- Riesgo de fuga de información entre las diferentes cuentas de clientes en caso de bugs en el sitio.<br><br>- Incapaz de proporcionar una copia de seguridad de su base de datos a un cliente sin proporcionar todos los datos.<br><br><br> \<br>\ Una base de datos por cuenta de cliente \ <br><br>- Alto nivel de seguridad contra fugas de datos: cada cuenta de cliente sólo puede acceder a su propia base de datos. Cuando se utiliza HFSQL en modo Cliente/Servidor, esta seguridad aumenta con la creación de una cuenta HFSQL para cada cuenta de cliente SaaS.<br><br>- Posibilidad de guardar la base de datos de cada cliente por separado y de proporcionar las copias de seguridad.<br><br>- El mantenimiento de una base de datos de clientes no afecta a las demás.<br><br>- La programación se simplifica porque no hay necesidad de diferenciar entre los registros de los diferentes clientes de la base de datos.<br><br><br> \ <br><br>- No se puede acceder a los datos de una cuenta de cliente desde otro sitio.<br><br><br> \<br><br> |
| *Parámetros para acceder a los datos* | Según el modo de gestión de datos seleccionado, se debe especificar:<br><br>- **en el caso de una base de datos HFSQL Client/Server**: los parámetros para conectarse al servidor HFSQL así como el nombre de la base de datos reference (opcional) que se duplicará para crear cada base de datos de cuentas de clientes.<br><br>- **en el caso de una base de datos HFSQL Classic**: el nombre del directorio de los archivos de datos de reference.<br>	**Observación**: los directorios de datos de las diferentes cuentas de clientes se encuentran en el servidor de aplicaciones que aloja el sitio, en el directorio de datos de la cuenta WEBDEV propietaria del sitio.<br><br><br> |
