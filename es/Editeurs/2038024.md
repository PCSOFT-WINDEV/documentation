


## SCM: Política de reintegración
			



<a name="NOTE1"></a>
<a name="NOTE1_1"></a>


## Presentación
<a name="presentacion_ELTTEXTE000182"></a>
El SCM le da la posibilidad de definir una política de facturación. Simplemente defina los criterios que deben coincidir para que un elemento de proyecto vuelva a ser verificado en el repositorio.. Entre los posibles criterios:

- Ningún error de compilación, 

- Realice al menos una prueba del elemento, ....




Estas reglas pueden ser obligatorias o no. Si no se cumple una regla, el promotor puede tener que dar una explicación.

<a name="NOTE2"></a>
<a name="NOTE2_1"></a>


## ¿Cómo proceder?
<a name="¿como_proceder_ELTTEXTE000206"></a>


### Definición de una política de facturación
<a name="definicion_una_politica_facturacion_ELTPARAGRAPHE000023"></a>

Para definir una política de facturación: 

1. En la pestaña "SCM", en el grupo "Repositorio", despliegue "Administrar" y seleccione "Política de reintegración".<br>![](https://doc.pcsoft.fr/en-US/images/image.awp?langid=7&name=GDS_Politique.gif)


2. Seleccione las condiciones que deben cumplir los elementos facturados. Estas reglas se agrupan en tres pestañas: "General", "Pruebas automatizadas" y "Métrica del código". 

3. Especifique las opciones de la política de facturación:

	- Ignorar siempre las reglas del proyecto (wdp): Se usa para ignorar las reglas al manejar el archivo de proyecto (archivo wdp).

	- Capacidad para ignorar las reglas.

	- Requiere comment si no se cumple una condición.




4. Validar. La política de facturación se tiene en cuenta automáticamente para el proyecto actual.




**Observación**: La política de registro también puede ser definida por el administrador de SCM ("Política de registro" de el menú contextual de un proyecto).
<a name="NOTE2_2"></a>


### Cómo funciona la política de facturación
<a name="como_funciona_politica_facturacion_ELTPARAGRAPHE000056"></a>

Si el elemento facturado no cumple con la política de facturación definida para el proyecto, aparece la siguiente ventana:<br>![](https://doc.pcsoft.fr/en-US/images/image.awp?langid=7&name=GDS_NonRespect.gif)
 Esta ventana enumera los criterios que no cumple el elemento facturado.

Si los usuarios pueden ignorar las reglas, pueden seleccionar la opción "Reintegrar de todos modos".

Si se requiere una comment, el botón de validación de esta ventana se desgrasará sólo cuando se introduzca la comment en el campo correspondiente.

El botón "Editar la política" le permite editar y modificar (si es necesario) las reglas definidas para la política de facturación.
<a name="NOTE2_3"></a>


### Almacenamiento y gestión de la política de facturación
<a name="almacenamiento_gestion_politica_facturacion_ELTPARAGRAPHE000075"></a>

La política de facturación se almacena en el archivo "CheckInPolicy.scm" que se encuentra en la raíz del directorio del proyecto.. Este archivo se encuentra en la SCM como todos los archivos del proyecto. Este archivo puede ser compartido entre varios proyectos.

Se pueden asignar derechos específicos al archivo correspondiente a la política de facturación (para que el director de proyecto pueda modificar este archivo, por ejemplo).

Si el archivo "CheckInPolicy.scm" se encuentra en el directorio principal, todos los proyectos recién creados utilizarán esta política de registro.

Consejo: Para una sucursal, puede definir una política de facturación más restrictiva.

## Detalles de las diferentes normas
<a name="detalles_las_diferentes_normas_ELTTEXTE000242"></a>


### Pestaña "General"
<a name="pestana_general_ELTPARAGRAPHE000090"></a>

Las diferentes reglas de facturación propuestas por la pestaña "General" son:

- Ningún error de compilación: Para ser verificado, el elemento no debe incluir errores de compilación.

- Ninguna advertencia de compilación: Para ser registrado, el elemento no debe incluir advertencias de compilación.

- Ninguna información de compilación: Para ser registrado, el elemento no debe incluir ninguna información de compilación.

- Ningún error de estándar de programación: Para ser verificado, el elemento no debe incluir errores estándar de programación.. Este estándar de programación se define en la pestaña "Compilación" de la ventana del proyecto description. Para obtener más información, consulte [Descripción del proyecto: Ficha Compilación](../Editeurs/9500218.md). 

- El elemento registrado debe ser associated con una tarea o con un incidente.

- Se debe introducir una comment durante la operación de registro.

- La prueba del proyecto (GO) debe haberse ejecutado al menos una vez después de una modificación.



<a name="NOTE3_2"></a>


### "Pestaña "Pruebas automatizadas
<a name="pestana_pruebas_automatizadas_ELTPARAGRAPHE000110"></a>

Las reglas de registro propuestas por la pestaña "Pruebas automatizadas" son:

- Cada elemento debe tener al menos 1 prueba automatizada. Atención: si se selecciona esta opción, se aplica a todos los elementos, incluso al archivo de proyecto (".WDP") que contiene el código de inicialización del proyecto.

- No hay pruebas automatizadas con errores.



<a name="NOTE3_3"></a>


### "Ficha"Métricas de código
<a name="fichametricas_codigo_ELTPARAGRAPHE000120"></a>

La pestaña "Code metrics" (métricas de código) se utiliza para introducir el nivel de porcentaje mínimo que se debe respetar para los comentarios de código.
<a name="NOTE4_3"></a>


### Pestaña "Plan de acción"
<a name="pestana_plan_accion_ELTPARAGRAPHE000127"></a>

En esta pestaña, puede definir el plan de acción que se ejecutará cuando se registre el proyecto. Si se va a ejecutar un plan de acción, especifique: 

- el coordinador de la fábrica de software,

- el puerto del coordinador,

- el usuario y la contraseña de associated. 


Puede elegir entre los planes existentes en el coordinador.
