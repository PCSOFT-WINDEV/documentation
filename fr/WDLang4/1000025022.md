


## &lt;Source&gt;.LitPremier (Fonction)

***En anglais : &lt;Source&gt;.ReadFirst***



<a name="XUtilisation"></a>
<a name="Utilisation"></a>
<a name="description"></a>
Positionne sur le premier enregistrement d'un fichier de données en fonction d'une rubrique de parcours. L'enregistrement est lu et les variables HFSQL (par exemple Client.Nom, c'est-à-dire la rubrique *Nom* du fichier *Client*) sont mises à jour.

La lecture se fait de la plus petite à la plus grande valeur de la rubrique de parcours (consultez les remarques pour plus de détails).

En général, la fonction **&lt;Source&gt;.LitPremier** permet de se positionner dans le fichier de données pour effectuer une boucle de lecture avec la fonction [&lt;Source&gt;.LitSuivant](../WDLang4/1000025035.md).

Après l'exécution de la fonction **&lt;Source&gt;.LitPremier**, plusieurs cas peuvent se présenter :

- le fichier de données est vide ou il n'existe pas d'enregistrement correspondant au filtre (défini par [&lt;Source&gt;.Filtre](../WDLang4/1000024960.md)) : aucune lecture n'est effectuée et la fonction [&lt;Source&gt;.EnDehors](../WDLang4/1000024605.md) renvoie <u><u><u>Vrai</u></u></u>.

- la fonction tente de lire un enregistrement déjà bloqué en lecture : aucune lecture n'est effectuée, la fonction [HErreurBlocage](../WDLang4/3044099.md) renvoie <u><u><u>Vrai</u></u></u> et la fonction [&lt;Source&gt;.EnDehors](../WDLang4/1000024605.md) renvoie <u><u><u>Vrai</u></u></u>.
	
	![Java](https://doc.pcsoft.fr/ext/images/fr/JAVA.png) Accès par JDBC : La gestion des blocages n'est pas disponible sur les bases de données accédées par JDBC.




Cette fonction peut être utilisée indifféremment avec les fichiers de données, les vues HFSQL ou les requêtes.

<a name="Exemple1"></a>
<a name="sample_code"></a>

## Exemple


```wl
Client.LitPremier(Nom)
TANTQUE Client.EnDehors() = Faux
	// Traitement de l'enregistrement
	Client.LitSuivant(Nom)
FIN
```

<a name="SYNTAXE1"></a>

## Syntaxe

`<Résultat> = <Source>.LitPremier([<Rubrique de parcours> [, <Options>]])`
---

**`<Résultat> : (Booléen)`**

Correspond à : 

- <u><u><u><u>Faux</u></u></u></u> en cas d'erreur. Dans ce cas, la fonction [HErreur](../WDLang4/3044088.md) renvoie un entier différent de 0. La fonction [HErreurInfo](../WDLang4/3044071.md) permet d'obtenir plus de détails sur l'erreur. L'enregistrement n'est pas lu. 

- la valeur de la fonction [&lt;Source&gt;.Trouve](../WDLang4/1000025076.md) dans les autres cas (l'enregistrement peut être lu, même si &lt;Résultat&gt; renvoie <u><u><u><u>Faux</u></u></u></u>).




**`<Source> : (Type correspondant à la source spécifiée)`**

Nom du fichier de données, de la vue HFSQL ou de la requête manipulé. 

**`<Rubrique de parcours> : (Chaîne de caractères optionnelle)`**

Nom de la rubrique utilisée pour parcourir le fichier de données ou la vue (ce paramètre n'est pas pris en compte pour les requêtes). Si ce nom n'est pas spécifié, la fonction **&lt;Source&gt;.LitPremier** va utiliser :

- ***Dans le cas d'un fichier de données*** : la dernière rubrique de parcours utilisée sur ce fichier par la dernière fonction de gestion HFSQL (fonction commençant par la lettre H). Si cette rubrique n'existe pas, la meilleure rubrique de parcours est automatiquement utilisée. 

- ***Dans le cas d'une requête*** : le ORDER BY de la requête si il existe, sinon selon la dernière rubrique utilisée. 

- ***Dans le cas d'une vue*** : la rubrique de tri de la vue (si elle existe), sinon sur la dernière rubrique utilisée.




**`<Options> : (Constante optionnelle)`**

Permet de paramétrer :

- le blocage réalisé sur l'enregistrement lu par la fonction **&lt;Source&gt;.LitPremier**

- la prise en compte ou non du filtre précédemment défini.

## Remarques


### Rubrique de parcours
<a name="rubrique_parcours_ELTPARAGRAPHE000508"></a>Si la rubrique de parcours utilisée est une clé, la fonction **&lt;Source&gt;.LitPremier** lit l'enregistrement pour lequel la valeur de la clé est la plus petite. L'ordre de tri pris en compte est l'ordre de tri spécifié dans l'analyse pour cette clé. En cas de doublons, **&lt;Source&gt;.LitPremier** lit le premier enregistrement "doublon" selon l'ordre des numéros d'enregistrement.

Si la rubrique de parcours n'est pas une clé, la fonction **&lt;Source&gt;.LitPremier** lit le premier enregistrement actif. En cas de parcours du fichier, les enregistrements seront triés selon leur numéro d'enregistrement. 
Dans ce cas, la rubrique de parcours sélectionnée apparaîtra en rouge sous l'éditeur de code et un warning sera affiché dans le [volet "Code"](../Editeurs/2027009.md).

**Remarque** : La [complétion automatique](../Editeurs/2013001.md) propose uniquement les rubriques clés.

### Parcours des requêtes
<a name="parcours_des_requetes_ELTPARAGRAPHE000541"></a>Par défaut, la fonction **&lt;Source&gt;.LitPremier** ré-exécute la requête pour rafraîchir le résultat de la requête. Il est conseillé d'utiliser la constante *hSansRafraîchir* pour éviter la ré-exécution de la requête.

**Parcours d'une requête exécutée avec la constante hRequêteSansCorrection** :

Pour parcourir les enregistrements dans l'ordre renvoyé par la base de données, il n'est pas nécessaire de préciser de rubrique de parcours. Exemple :

```wl
MaRequête.ExécuteRequête(hRequêteSansCorrection)
...
MaRequête.LitPremier(hSansRafraîchir)
```
Si une rubrique de parcours est spécifiée, le résultat de la requête est intégralement récupéré et indexé. Le parcours est alors réalisé sur la rubrique spécifiée. Le tri initial de la requête (spécifié par ORDER BY) est ignoré. L'index créé (au format HFSQL) est sensible à la casse, sensible à la ponctuation, sensible aux accents et ascendant.

Exemple :

```wl
MaRequête.ExécuteRequête(hRequêteSansCorrection)
...
MaRequête.LitPremier(MaRubrique, hSansRafraîchir)
```
Les recherches utilisent l'index créé sur le résultat de la requête.

### Blocages
<a name="blocages_ELTPARAGRAPHE000576"></a>Par défaut (paramètre &lt;Option&gt; non spécifié), aucun blocage de l'enregistrement n'est réalisé.

Si un blocage est demandé (constantes *hBlocageEcriture* ou *hBlocageLectureEcriture*), la lecture de l'enregistrement sera réalisée uniquement si cet enregistrement n'est pas déjà bloqué.

![OLE DB](https://doc.pcsoft.fr/ext/images/fr/OLEDB.png)![Connecteurs Natifs (Accès Natifs)](https://doc.pcsoft.fr/ext/images/fr/AN.png) Si le provider OLE DB ou le Connecteur Natif utilisé ne gère pas les blocages, les options de blocage seront sans effet.

### Mémos
<a name="memos_ELTPARAGRAPHE000592"></a>Les mémos associés à l'enregistrement peuvent être automatiquement lus ou non lors de la lecture de l'enregistrement. La fonction [&lt;Source&gt;.GèreMémo](../WDLang4/1000024987.md) permet de personnaliser cette lecture automatique.

Si les mémos sont gérés, lorsque l'enregistrement est lu, les mémos texte associés sont également lus. Les mémos binaires ne sont lus que lorsqu'ils sont utilisés explicitement (fonction [&lt;Source&gt;.ExtraitMémo](../WDLang4/1000024957.md)).
<a name="NOTE0_5"></a>


### Mot de passe
<a name="mot_passe_ELTPARAGRAPHE000614"></a>Si la fonction **&lt;Source&gt;.LitPremier** est la première fonction manipulant le fichier de données spécifié, la vérification du mot de passe est effectuée lors de l'ouverture du fichier de données. Si le mot de passe est incorrect, la fonction [HErreurMotDePasse](../WDLang4/3044082.md) renvoie <u><u><u><u>Vrai</u></u></u></u> et la fonction [&lt;Source&gt;.EnDehors](../WDLang4/1000024605.md) renvoie <u><u><u><u>Vrai</u></u></u></u>.

### Connecteur Natif XML
<a name="connecteur_natif_xml_ELTPARAGRAPHE000645"></a>Le comportement de la fonction **&lt;Source&gt;.LitPremier** est lié aux fonctions [&lt;Source&gt;.ActiveFiltreAuto](../WDLang4/1000024171.md)/[&lt;Source&gt;.DésactiveFiltreAuto](../WDLang4/1000024219.md).

Par défaut, la fonction [&lt;Source&gt;.ActiveFiltreAuto](../WDLang4/1000024171.md) est activée.

Ainsi, pour lire le contenu d'un document XML, il suffit de lire le contenu du fichier principal (le parent) puis des fichiers liés (les fils).

Par défaut, lors de la lecture dans un fichier de données, un filtre est automatiquement réalisé sur les fichiers de données reliés pour lire uniquement les enregistrements correspondant au fichier principal.

Par exemple :
![](https://doc.pcsoft.fr/fr-FR/images/image.awp?langid=5&name=XML_2.GIF)
Lors du parcours du fichier Person, il est possible de connaître l'email de cette personne.

Pour cela, il suffit de se positionner sur le fichier "Person", puis d'utiliser la fonction **&lt;Source&gt;.LitPremier** sur le fichier "email".

Dans ce cas, l'enregistrement lu dans le fichier "email" correspondra au premier email associé à l'enregistrement en cours dans le fichier "Person".

Si ce mécanisme est désactivé (fonction [&lt;Source&gt;.DésactiveFiltreAuto](../WDLang4/1000024219.md)), l'enregistrement lu dans le fichier "email" correspondra au premier enregistrement du fichier email (et non au fils de l'enregistrement lu dans le fichier "Person").

### Divers
<a name="divers_ELTPARAGRAPHE000697"></a>

- Le numéro de l'enregistrement en cours est connu avec la fonction [&lt;Source&gt;.NumEnr](../WDLang4/1000025041.md).

- La fonction [&lt;Source&gt;.ChangeClé](../WDLang4/1000024198.md) permet de modifier la clé de parcours, tout en restant positionné sur l'enregistrement en cours.

- Pour optimiser le temps des premiers parcours d'un fichier, utilisez la fonction [&lt;Source&gt;.Optimise](../WDLang4/1000025042.md) sur ce fichier de données.

- Cette fonction remplace les fonctions **HLitPremierBloque** et **HLitPremierSansBloquer**, conservées par compatibilité avec WINDEV 5.5.
