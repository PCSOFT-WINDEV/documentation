
## Administrateur SaaS - Ajouter un site
			



## Présentation
<a name="presentation_ELTTEXTE000128"></a>
L'ajout d'un site SaaS permet de déclarer un site SaaS déployé sur un Serveur d'Application WEBDEV à un Webservice SaaS afin de permettre son utilisation.

Cette opération doit être réalisée après le déploiement du site.

**Remarque** : Cette opération est équivalente à l'utilisation de la fonction [SaaSAdminAjouteSite](../WDLang3/1000019102.md).







## Comment le faire ?
<a name="comment_faire_ELTTEXTE000158"></a>
Pour ajouter un site SaaS, depuis le menu principal de l'Administrateur SaaS, sélectionnez l'option "Sites .. Ajouter un site".

**Rappel** : L'Administrateur SaaS est situé sur le Serveur d'Application hébergeant le Webservice SaaS qui contrôle le site. 

Les éléments suivants sont demandés :


|   |   |
| --- | --- |
| **Elément** | **Rôle** |
| *Nom du site* | Il s'agit du nom de déploiement du site WEBDEV.<br>Ce nom a été fourni lors de la procédure de déploiement du site dans l'assistant d'installation.<br>Par défaut, il s'agit du nom du projet.<br><br>**Remarque** : Ce paramètre est sensible à la casse. |
| *Mode de gestion des données* | Plusieurs choix sont possibles :<br><br>\ Mode de gestion des données \ Avantages \ Inconvénients \<br>\ --- \ --- \ --- \<br>\ Pas de base de données \   \   \<br>\ Une base de données unique partagée par tous les comptes clients \ <br><br>- Possibilité de mutualiser des éléments entre les différents clients (si le site utilise des mémos volumineux, ils n'ont pas à être recopiés pour chaque client).<br><br>- Simplification de la maintenance de la base de données : une seule sauvegarde à réaliser, une seule modification automatique en cas d'évolution de l'analyse du site.<br><br><br> \ <br><br>- Risque de fuite d'informations entre les différents comptes clients en cas de bug dans le site.<br><br>- Impossibilité de livrer une sauvegarde de sa base à un client sans fournir la totalité des données.<br><br><br> \<br>\ Une base de données par compte client \ <br><br>- Grande sécurité contre la fuite d'informations : chaque compte client ne peut accéder qu'à sa base. Dans le cas de l'utilisation d'une base HFSQL en mode Client/Serveur, cette sécurité est encore accrue par la création d'un compte HFSQL pour chaque compte client SaaS.<br><br>- Possibilité de sauvegarder séparément la base de données de chaque client et de livrer les sauvegardes.<br><br>- La maintenance de la base d'un client n'a pas d'incidence sur les autres.<br><br>- La programmation est simplifiée puisqu'il n'est pas nécessaire de différencier les enregistrements des différents clients dans la base.<br><br><br> \ <br><br>- Impossibilité d'accéder aux données d'un compte client depuis un autre site<br><br><br> \<br><br> |
| *Paramètres d'accès aux données* | Selon le mode de gestion des données choisi, il convient d'indiquer :<br><br>- **dans le cas d'une base HFSQL Client/Serveur** : les paramètres de connexion au serveur HFSQL ainsi que le nom de la base de référence (optionnelle) qui sera dupliquée pour créer chaque base des comptes clients.<br><br>- **dans le cas d'une base HFSQL Classic** : le nom du répertoire des fichiers de données de référence.<br>	**Note** : les répertoires de données des différents comptes clients sont localisés sur le Serveur d'Application hébergeant le site, dans le répertoire des données du compte WEBDEV propriétaire du site.<br><br><br> |
